    从输入网址到看到网页：一次完整的网络之旅

    你有没有想过，当你在浏览器输入 www.baidu.com 并按下回车后，短短几百毫秒内，究竟发生了什么？数据是如何从你的电脑，穿越千山万水，到达百度的服务器，又如何带回你想要的网页？

    今天，让我们一起揭开这个过程的神秘面纱。

    准备工作：计算机网络的五层模型

    在开始我们的旅程之前，先了解一下计算机网络的基本结构。整个网络系统可以分为五层，每一层负责不同的工作：

    ┌─────────────────────────┐
│   应用层                 │  你看到的：网页、视频、邮件
│   HTTP、DNS、SMTP...     │
├─────────────────────────┤
│   传输层                 │  端到端：可靠还是快速？
│   TCP、UDP              │
├─────────────────────────┤
│   网络层                 │  全局寻址：找到目的地
│   IP、路由              │
├─────────────────────────┤
│   数据链路层             │  局部转发：同一网络内
│   MAC地址、交换机        │
├─────────────────────────┤
│   物理层                 │  物理传输：网线、光纤
│   电信号、光信号         │
└─────────────────────────┘

    核心思想：

    • 分层：每层只关心自己的事，互不干扰
    • 封装：一层套一层，就像俄罗斯套娃
    • 端到端：中间的路由器、交换机只负责转发，不管你的网页内容

    好，现在让我们开始这场旅程！

    第一站：应用层 —— DNS查询

    场景：你在浏览器输入 www.baidu.com

    浏览器收到这个域名后，第一件事不是立即发送请求，而是要先把这个”人类能看懂的域名”翻译成”计算机能理解的IP地址”。

    DNS查询的优先级：

    1. 浏览器缓存
   → "我之前访问过百度吗？"
   → 有缓存 → 直接用
   
2. 操作系统缓存
   → "系统之前查过吗？"
   
3. Hosts文件（本地DNS配置）
   → Windows: C:\Windows\System32\drivers\etc\hosts
   → Linux/Mac: /etc/hosts
   
4. 都没有 → 向本地DNS服务器查询
   → 这个DNS服务器通常是你的ISP提供的
   → 或者你手动配置的（如8.8.8.8，Google的DNS）

    DNS的工作方式：递归 + 迭代

    这里有个关键点要理解：

    你的电脑 ↔ 本地DNS服务器：递归查询
（你只问一次，等答案）

本地DNS服务器 ↔ 各级DNS服务器：迭代查询
（本地DNS自己一级级问）

    完整过程：

    你：            "www.baidu.com的IP是什么？"
                    ↓
本地DNS服务器：  "让我帮你查！"
                    ↓
本地DNS → 根DNS：      "www.baidu.com在哪？"
根DNS → 本地DNS：      "我不知道，但.com的DNS在这里：x.x.x.x"
                    ↓
本地DNS → .com DNS：   "www.baidu.com在哪？"
.com DNS → 本地DNS：   "我不知道，但baidu.com的DNS在这里：y.y.y.y"
                    ↓
本地DNS → baidu.com DNS： "www.baidu.com的IP是什么？"
baidu.com DNS → 本地DNS： "是 220.181.38.148"
                    ↓
本地DNS → 你：         "查到了！是 220.181.38.148"

    注意：

    • DNS查询使用 UDP协议（传输层）
    • 为什么用UDP？因为查询数据量小（几十字节），一个包就够了，不需要TCP的复杂握手
    • 如果UDP丢包了，客户端会重发，总时间也比TCP快

    可能的CDN介入：

    如果百度使用了CDN（内容分发网络），DNS查询会更有意思：

    你查询：www.baidu.com

DNS返回：
第1步：CNAME记录 → www.baidu.com.cdn.cloudflare.com
第2步：根据你的地理位置，智能返回离你最近的CDN节点IP
       - 你在北京 → 返回北京CDN节点：1.2.3.4
       - 朋友在上海 → 返回上海CDN节点：5.6.7.8

结果：你访问的是离你最近的CDN节点，而不是百度的源站！
速度快得多！

    第二站：传输层 —— TCP三次握手

    现在我们知道了百度的IP地址：220.181.38.148，可以开始建立连接了。

    因为我们要请求网页，需要确保数据完整、可靠，所以使用 TCP协议（而不是UDP）。

    TCP三次握手：建立连接

    TCP是面向连接的协议，通信前必须先”握手”：

    你的电脑                             百度服务器
  (客户端)                           (服务器)
    |                                   |
    | ------ SYN, Seq=1000 -----------> |  第1次握手
    |        "我想连接你"                |  "我想连接你"
    |                                   |
    | <-- SYN+ACK, Seq=2000 ----------- |  第2次握手
    |        Ack=1001                    |  "我也想连接你，我收到你的请求了"
    |                                   |
    | ---- ACK, Ack=2001 -------------> |  第3次握手
    |        "我收到你的回复了！"         |  "连接建立！"
    |                                   |
    |=========== 连接建立 ==============|

    为什么需要三次握手，而不是两次？

    举个例子：

    两次握手的问题：

你：   "喂，听得到吗？"（第1次）
对方： "听得到！"（第2次）
然后对方就开始balabala说一大堆...

但你根本没听到对方的回复（网络延迟了）！
你以为没人接，已经挂了。
对方在那自嗨半天，白费力气！

三次握手：

你：   "喂，听得到吗？"（第1次）
对方： "听得到！你听得到我吗？"（第2次）
你：   "听得到！咱们开始聊吧！"（第3次）

双方都确认了："你能听到我，我也能听到你！"

    如果是HTTPS（现在大部分网站都是）：

    TCP三次握手后，还要进行 TLS握手：

    1. ClientHello：客户端发送支持的加密算法列表
2. ServerHello：服务器选择加密算法，发送证书
3. 客户端验证证书（确认这真的是百度，不是钓鱼网站）
4. 密钥交换
5. 加密通道建立

之后所有HTTP数据都是加密的！
路由器、ISP都看不到你访问的具体内容！

    HTTP请求

    连接建立后，浏览器发送HTTP请求：

    GET / HTTP/1.1
Host: www.baidu.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) Chrome/120.0.0.0
Accept: text/html,application/xhtml+xml
Accept-Language: zh-CN,zh;q=0.9
Connection: keep-alive
Cookie: BAIDUID=xxx; BIDUPSID=xxx

    这个HTTP请求会被TCP分成若干个段（如果数据大的话），每个段都有序列号，确保按顺序到达。

    第三站：网络层 —— 寻找目的地

    现在，数据要从你的电脑（假设IP是 192.168.1.10）发送到百度服务器（220.181.38.148）。

    第一步：判断是否在同一子网

    网络层首先要判断：百度的服务器和我在同一个局域网吗？

    怎么判断？用子网掩码！

    你的电脑：
IP地址：192.168.1.10
子网掩码：255.255.255.0（通常DHCP自动分配的）

百度服务器：
IP地址：220.181.38.148

判断过程：
你的网络号 = 192.168.1.10 & 255.255.255.0 = 192.168.1.0
百度的网络号 = 220.181.38.148 & 255.255.255.0 = 220.181.38.0

192.168.1.0 ≠ 220.181.38.0
→ 不在同一子网！
→ 需要通过路由器（默认网关）转发

    什么是子网掩码？

    子网掩码是用来划分IP地址的”网络号”和”主机号”的：

    子网掩码：255.255.255.0
二进制：  11111111.11111111.11111111.00000000
         ↑ 网络号部分全是1 ↑ ↑主机号部分全是0↑

IP地址与子网掩码做"与运算"：
192.168.1.10  = 11000000.10101000.00000001.00001010
255.255.255.0 = 11111111.11111111.11111111.00000000
相与结果 = 11000000.10101000.00000001.00000000
          = 192.168.1.0（网络号）

这就是"截出网络号"的过程！

    第二步：找默认网关的MAC地址

    既然不在同一子网，数据包要先发给默认网关（通常是 192.168.1.1，你家的路由器）。

    但是，网络层只有IP地址，数据链路层需要MAC地址！

    这时候就要用到 ARP协议（地址解析协议）：

    第1步：查ARP缓存表
你的电脑查看本地ARP表："192.168.1.1的MAC地址是？"
没有记录 → 需要发ARP请求

第2步：发送ARP请求（广播）
你的电脑广播：
  源MAC：AA:AA:AA:AA:AA:AA（你的网卡MAC）
  目标MAC：FF:FF:FF:FF:FF:FF（广播地址）
  内容："请问192.168.1.1的MAC地址是什么？
        我是192.168.1.10，我的MAC是AA:AA:AA:AA:AA:AA"

第3步：路由器回复（单播）
路由器收到后：
  源MAC：AA:11:22:33:44:55（路由器的MAC）
  目标MAC：AA:AA:AA:AA:AA:AA（你的MAC）
  内容："我是192.168.1.1，我的MAC是AA:11:22:33:44:55"

第4步：更新ARP表
你的ARP缓存表：
+-----------------+----------------------+
| IP地址          | MAC地址              |
+-----------------+----------------------+
| 192.168.1.1     | AA:11:22:33:44:55    |
+-----------------+----------------------+

以后要发给网关，直接查表就行，不用重复ARP请求！
（但ARP表有过期时间，通常几分钟）

    第三步：NAT地址转换

    现在数据包准备离开你的家庭网络，进入互联网。

    问题来了：

    • 你的IP是 192.168.1.10（私有IP，内网专用）
    • 互联网上不认识这个IP！
    • 怎么办？

    解决方案：NAT（网络地址转换）

    你家的路由器会把你的内网IP转换成公网IP：

    出去的数据包（经过路由器WAN口时）：

转换前：
源IP：192.168.1.10（内网IP）
源端口：54321
目标IP：220.181.38.148（百度）
目标端口：80

路由器NAT转换：
源IP：192.168.1.10 → 123.45.67.89（你家的公网IP）
源端口：54321 → 10001（路由器可能修改端口）

转换后：
源IP：123.45.67.89（公网IP）
源端口：10001
目标IP：220.181.38.148
目标端口：80

路由器的NAT表记录：
+------------------------+--------------------+
| 内网地址               | 公网地址           |
+------------------------+--------------------+
| 192.168.1.10:54321     | 123.45.67.89:10001 |
+------------------------+--------------------+

    百度回复时的反向转换：

    百度的回复包到达你家路由器：
目标IP：123.45.67.89（你家公网IP）
目标端口：10001

路由器查NAT表：
10001端口 → 对应内网的 192.168.1.10:54321

反向转换：
目标IP：123.45.67.89 → 192.168.1.10
目标端口：10001 → 54321

转发给你的电脑！

    为什么需要NAT？

    因为IPv4地址不够用了！

    IPv4地址总数：2^32 = 约42亿个
全球设备数：几百亿

解决方案：
- 公网IP：分配给每个家庭/公司一个（贵！稀缺！）
- 内网IP：每个家庭/公司内部用私有IP（免费！无限！）
- NAT：让多台设备共享一个公网IP上网

你家可能有：
- 3台手机
- 2台电脑
- 1台平板
- 1个智能电视
→ 7台设备，但只用1个公网IP！

    第四站：数据链路层 —— 一跳一跳转发

    现在数据包准备从你的电脑发出去了。数据链路层负责在”同一个网络内”传输数据。

    封装以太网帧

    数据包现在被封装成这样：

    +----------------------------------+
|  以太网头部（数据链路层）          |
|  源MAC：AA:AA:AA:AA:AA:AA (你)    |
|  目标MAC：AA:11:22:33:44:55 (网关) |
+----------------------------------+
|  IP头部（网络层）                 |
|  源IP：192.168.1.10               |
|  目标IP：220.181.38.148           |
+----------------------------------+
|  TCP头部（传输层）                |
|  源端口：54321                    |
|  目标端口：80                     |
+----------------------------------+
|  HTTP数据（应用层）               |
|  GET / HTTP/1.1                   |
|  Host: www.baidu.com              |
+----------------------------------+

    交换机的工作

    数据帧到达交换机：

    交换机收到数据帧（从端口1进入）：

第1步：学习源MAC地址
"这个包从端口1进来，源MAC是AA:AA:AA:AA:AA:AA"
→ 记录到MAC地址表：
+------------------------+--------+
| MAC地址                | 端口   |
+------------------------+--------+
| AA:AA:AA:AA:AA:AA      | 1      |
+------------------------+--------+

第2步：查找目标MAC地址
"目标MAC是AA:11:22:33:44:55，在哪个端口？"
→ 查MAC地址表
→ 找到了：端口2
→ 从端口2转发出去

如果没找到：
→ 广播：从所有端口转发（除了端口1）
→ 路由器收到后会回复
→ 交换机学习路由器的MAC地址

    交换机只记录源MAC，不记录目标MAC！

    为什么？因为：

    • 数据包从端口1进来
    • 源MAC是AA:AA
    • 说明AA:AA这个设备在端口1那边 ✓
    • 目标MAC只是”要发给谁”，不代表”谁在哪”

    第五站：物理层 —— 物理传输

    数据帧最终变成物理信号：

    有线传输：
- 网线（双绞线）：电信号，1和0用不同电压表示
- 光纤：光信号，1和0用光的有无表示

无线传输：
- WiFi：无线电波（2.4GHz / 5GHz）
- 4G/5G：移动通信网络

    你的数据包通过网线，从电脑的网卡发出，经过交换机，到达路由器。

    第六站：路由器的转发 —— 跨网络传输

    路由器1（你家的路由器）

    路由器收到数据包：

第1步：拆掉以太网头部（数据链路层）
丢弃：源MAC（你的MAC）、目标MAC（路由器LAN口的MAC）
保留：IP头部、TCP头部、HTTP数据

第2步：NAT转换（我们前面讲过了）
源IP：192.168.1.10 → 123.45.67.89

第3步：查看目标IP
目标IP：220.181.38.148（百度）

第4步：查路由表
路由表：
+-------------------+--------------+--------+
| 目的网络          | 下一跳       | 端口   |
+-------------------+--------------+--------+
| 192.168.1.0/24    | -            | LAN    |
| 0.0.0.0/0 (默认)  | 202.1.1.1    | WAN    |
+-------------------+--------------+--------+

匹配到：默认路由 → 下一跳：202.1.1.1，从WAN口出去

第5步：封装新的以太网头部
查ARP：202.1.1.1的MAC地址 → BB:BB:BB:BB:BB:BB
源MAC：路由器WAN口的MAC（AA:22:33:44:55:66）
目标MAC：下一跳路由器的MAC（BB:BB:BB:BB:BB:BB）

第6步：从WAN口发出去

    关键点：IP地址不变，MAC地址每跳都变！

    第1跳：你的电脑 → 你家路由器
  源MAC：你的MAC        目标MAC：路由器LAN口MAC
  源IP：192.168.1.10   目标IP：220.181.38.148  ← IP不变！

NAT转换后，进入互联网：
  源IP：123.45.67.89   目标IP：220.181.38.148  ← IP还是不变！

第2跳：你家路由器 → ISP路由器
  源MAC：路由器WAN口MAC  目标MAC：ISP路由器MAC
  源IP：123.45.67.89   目标IP：220.181.38.148  ← IP还是不变！

第3跳：ISP路由器 → 骨干网路由器
  源MAC：ISP路由器MAC   目标MAC：骨干网路由器MAC
  源IP：123.45.67.89   目标IP：220.181.38.148  ← IP还是不变！

...

第N跳：最后一个路由器 → 百度服务器
  源MAC：路由器MAC     目标MAC：百度服务器MAC
  源IP：123.45.67.89   目标IP：220.181.38.148  ← IP从头到尾都没变！

    为什么IP地址不变，MAC地址每跳都变？

    形象的类比：

    IP地址 = 导航的起点和终点
  "从北京到上海"
  → 从出发到到达，这两个地点从不改变

MAC地址 = 当前路段
  第1段：你家 → 高速入口
  第2段：高速入口 → 服务区
  第3段：服务区 → 高速出口
  第4段：高速出口 → 目的地
  
  每一段的起点和终点都在变！
  但导航上的"北京→上海"从不改变！

    技术原因：

    MAC地址：
- 数据链路层的地址
- 只在"同一个网络"内有效
- 每经过一个路由器，就进入一个新网络
- 所以MAC地址要换成新网络的地址

IP地址：
- 网络层的地址
- 全局唯一，全球有效
- 端到端的地址，不需要改变
- 路由器根据IP地址决定往哪转发

    第七站：到达百度服务器

    经过十几个甚至几十个路由器的转发，数据包终于到达百度的服务器！

    逐层拆封

    百度服务器收到数据包：

物理层：
网卡接收到电信号 → 转换成数字信号

数据链路层：
检查目标MAC地址：是我的 ✓
→ 拆掉以太网头部

网络层：
检查目标IP地址：220.181.38.148，是我的 ✓
→ 拆掉IP头部

传输层：
检查目标端口：80，是我的Web服务 ✓
→ 拆掉TCP头部
→ 交给Web服务器程序

应用层：
Web服务器（如Nginx）收到HTTP请求：
GET / HTTP/1.1
Host: www.baidu.com

→ 处理请求（查数据库、生成HTML）
→ 返回HTTP响应

    服务器处理

    1. 解析HTTP请求
   "客户端要访问首页，使用HTTP/1.1协议"

2. 查找资源
   "读取 /var/www/baidu/index.html"

3. 生成HTTP响应
   HTTP/1.1 200 OK
   Content-Type: text/html; charset=utf-8
   Content-Length: 12345
   Server: nginx/1.20.1
   
   <!DOCTYPE html>
   <html>
   <head>
       <title>百度一下，你就知道</title>
   </head>
   <body>
       ...
   </body>
   </html>

4. 发送响应
   响应数据被TCP分成多个段
   每个段都有序列号
   确保按顺序到达客户端

    第八站：原路返回

    响应数据沿着原路（或类似的路径）返回：

    百度服务器 
  → 骨干网路由器 
  → ISP路由器 
  → 你家路由器（反向NAT转换）
  → 交换机 
  → 你的电脑

每一跳：
- MAC地址都在变
- IP地址不变（百度IP → 你家公网IP）
- 到达你家路由器时，反向NAT转换：
  目标IP：123.45.67.89 → 192.168.1.10

    你的电脑收到响应

    传输层：
TCP检查序列号，重组数据
→ 发送ACK确认

应用层：
浏览器收到完整的HTML
→ 解析HTML
→ 发现需要加载CSS、JS、图片
→ 对每个资源重复上述过程
→ 渲染网页
→ 你看到了百度首页！

    第九站：四次挥手 —— 关闭连接

    通信完成后，TCP连接需要正式关闭：

    客户端                               服务器
   |                                   |
   | ---- FIN, Seq=m ----------------> |  第1次挥手
   |     "我没有数据要发了"             |  "我发完了"
   |                                   |
   | <--- ACK, Ack=m+1 --------------- |  第2次挥手
   |     "我知道了"                     |  "我知道你发完了"
   |     (服务器可能还有数据要发)        |
   |                                   |
   | <--- FIN, Seq=n ------------------ |  第3次挥手
   |     "我也没有数据了"               |  "我也发完了"
   |                                   |
   | ---- ACK, Ack=n+1 ---------------> |  第4次挥手
   |     "我知道了，再见！"              |  "再见！"
   |                                   |
   |======= 连接关闭 ===================|

    为什么是四次挥手，而不是三次？

    因为TCP是全双工通信：

    握手时（三次）：
- 双方同时同意建立连接
- 服务器可以把SYN和ACK合并发送（SYN+ACK）

挥手时（四次）：
- 客户端说："我发完了"（FIN）
- 服务器说："知道了，但我可能还没发完"（ACK）
- 服务器继续发送剩余数据...
- 服务器说："我也发完了"（FIN）
- 客户端说："知道了"（ACK）

所以必须是四次！

    完整流程图

    让我用一张图总结整个过程：

    【应用层】
输入网址 → DNS查询（递归+迭代）→ 获得IP

【传输层】
TCP三次握手 → (HTTPS: TLS握手) → 发送HTTP请求

【网络层】
判断子网 → 查ARP表 → NAT转换 → 查路由表

【数据链路层】
封装以太网帧 → 交换机转发

【物理层】
电信号/光信号传输

【经过N个路由器】
每个路由器：拆MAC → 查路由表 → 封装新MAC → 转发
IP地址不变，MAC地址每跳都变

【到达服务器】
逐层拆封 → 处理请求 → 发送响应

【原路返回】
响应数据返回 → 反向NAT → 到达你的电脑

【关闭连接】
TCP四次挥手

【浏览器渲染】
解析HTML → 加载资源 → 显示网页

    关键概念总结

    1. 三张重要的表

    MAC地址表（交换机）：
MAC地址 → 端口号
作用：在同一网络内转发

路由表（路由器）：
目的网络 → 下一跳 + 端口
作用：在不同网络间转发

ARP缓存表（主机和路由器）：
IP地址 → MAC地址
作用：IP地址解析为MAC地址

    2. 地址的层次

    域名（应用层）：www.baidu.com
      ↓ DNS解析
IP地址（网络层）：220.181.38.148
      ↓ 全局唯一，端到端不变
MAC地址（数据链路层）：AA:BB:CC:DD:EE:FF
      ↓ 局部有效，每跳都变
物理信号：010101...

    3. TCP vs UDP

    TCP：
✓ 面向连接（三次握手）
✓ 可靠传输（重传机制）
✓ 有序传输（序列号）
✓ 流量控制
✓ 拥塞控制
